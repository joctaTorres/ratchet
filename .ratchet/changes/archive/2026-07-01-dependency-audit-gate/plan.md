# dependency-audit-gate

## Why

The `ci-npx-release` batch publishes `ratchet` to npm only when every quality gate is green, and its promise is that "only when green" is a **real, unit-tested decision**, not just YAML wiring. Phase 1 shipped the dry-run pipeline skeleton plus the pure, unit-tested `decideRelease` spine and the thin `release-gate` runner (`src/core/ci/release-decision.ts`, `src/core/ci/release-gate.ts`). Phase 2 added the **coverage** and **e2e** gates the same way — a pure evaluator + thin runner per slice (`coverage-gate.ts`, `e2e-gate.ts`), then a wiring change that plugged both into the spine.

Phase 3 (`security-layer-gate`) adds a **security layer** as a hard gate: a CI job that runs a dependency vulnerability audit and a secret scan (and prepares npm provenance), wired into the release-decision module so a known vulnerability or a leaked secret flips the gate to **DENY**. That phase is sliced into three changes: this one (`dependency-audit-gate`), `secret-scan-gate`, and `wire-security-into-release-gate` (which runs `after` both and plugs their signal(s) into the spine and ships the phase proof `test/e2e/security-gate.sh`).

This change is the **dependency-audit slice**: a pure, exhaustively unit-tested evaluator that turns a parsed vulnerability audit into a `green`/`red` **gate signal** against a configured severity threshold, plus the CI wiring that runs the audit and goes red when a vulnerability at or above that severity is present. It deliberately mirrors the phase-1/phase-2 split — a pure module + a thin runner + a minimal CI step — and **does not** wire the signal into `decideRelease`'s wired-gate set. That wiring belongs to `wire-security-into-release-gate`. This slice exists to prove the "no vulnerability at/above the threshold" decision in isolation so the wiring change has a trustworthy signal to feed in.

## What Changes

- **Pure dependency-audit evaluator.** Add a shippable module under `src/core/ci/` (`src/core/ci/dependency-audit-gate.ts`) exporting a pure `evaluateDependencyAudit(input)` that takes parsed per-severity vulnerability counts (e.g. `{ info, low, moderate, high, critical }`) and a configured minimum severity to fail on, and returns an explicit `{ signal: 'green' | 'red'; counts: SeverityCounts | null; failOn: Severity; offending: string[]; reasons: string[] }`. `green` iff the counts parsed (non-null) AND **zero** vulnerabilities exist at any severity at or above `failOn`; otherwise `red` with a precise reason naming the offending severities/counts (or "audit report could not be read" when the counts are missing/malformed). The returned `signal` is exactly the `GateSignal` shape (`'green' | 'red'`) the release-decision module already consumes, so the later wiring change feeds it in unchanged.
- **Severity threshold as data.** Define a severity ordering (`info < low < moderate < high < critical`) and a default fail-on constant (e.g. `DEFAULT_AUDIT_FAIL_ON = 'high'`) overridable via an env var (e.g. `AUDIT_FAIL_ON`), so the value is configurable, not hardcoded at the call site. Default to failing on `high` and above — the conventional release-blocking bar — so low/moderate advisories do not turn a green tree red, while a high/critical vulnerability trips the gate.
- **Audit report parsing, fail-closed.** Add a small reader that extracts the per-severity counts from the package manager's machine-readable audit output (`pnpm audit --json` / `npm audit --json` writes a `metadata.vulnerabilities` block keyed by severity). A missing, unreadable, or malformed report yields `null`, which the evaluator treats as `red` — fail-closed, matching the spine's posture.
- **Thin audit-gate runner.** Add a thin, impure entrypoint (`runDependencyAuditGate(env)` in the same module, run when invoked directly à la the sibling gates' `isDirectRun()`) that resolves the report path + fail-on severity from the environment, reads the counts, calls `evaluateDependencyAudit`, prints the outcome and any reasons, and exits `0` on `green` / non-zero on `red`. It adds **no** decision logic — it only adapts CI's world to the pure evaluator, exactly as `release-gate.ts` / `coverage-gate.ts` / `e2e-gate.ts` do.
- **CI step.** Add a dependency-audit step to `.github/workflows/ci.yml`, positioned **after** the existing `test` step and **before/outside** the release path: run the audit writing its JSON report (e.g. `pnpm audit --json > <report> || true`, tolerating the audit's own non-zero exit so the gate runner — not the raw audit — owns the red/green decision), then `pnpm build` (if needed) and invoke the built audit-gate runner (`node dist/core/ci/dependency-audit-gate.js`) so a vulnerability at or above the threshold exits non-zero and turns the run red. Give the step an `id` (e.g. `audit`) the later wiring change can read. Do **not** add a security signal to the release-gate step and do **not** extend `WIRED_GATES` — that is `wire-security-into-release-gate`'s job.
- **Tests** under `test/ci/` (so they run with the suite):
  - `test/ci/dependency-audit-gate.test.ts`: `green` on zero vulnerabilities and on only below-threshold severities; `red` when a vulnerability at or above the threshold is present (reason names the severity/count); fail-closed `red` on a missing/malformed report (reason names the unreadable report); fail-on override respected (e.g. `critical` lets `high` pass); and the returned `signal` is assignable to the release-decision `GateSignal` type.
  - Audit CI-step assertions (using the existing `test/ci/helpers/workflow.ts` parser): a step runs the dependency audit and writes a JSON report; an audit-gate step invokes the evaluator and sits after the `test` step; and the release path still wires only `lint`/`test`/`coverage`/`e2e` (no security signal yet).
- Implements `features/ci/dependency-audit-gate.feature` and `features/ci/audit-ci-step.feature`.
- **Out of scope (explicitly):** the secret scan (`secret-scan-gate`), npm provenance preparation, wiring the security signal into `decideRelease`, and the phase proof harness `test/e2e/security-gate.sh` — all belong to the sibling changes (`secret-scan-gate`, `wire-security-into-release-gate`). The publish path remains `--dry-run`.
- Multi-agent surface: **none**. A pure evaluator, a thin runner, workflow YAML, and tests; no agent-specific files, skills, or commands.

## Design

**Thin vertical slice, same spine.** The phase goal is "a known vulnerability OR a leaked secret flips the gate to DENY". The full proof (`bash test/e2e/security-gate.sh`) spans all three phase-3 changes. This change owns the smallest end-to-end dependency-audit piece: run the audit → parse the report → decide green/red against a severity threshold → emit a gate signal in the spine's shape, with CI going red on a vulnerable tree. Joining that signal to `decideRelease` (extending `WIRED_GATES`, feeding the security signal) is the explicit `after` change — kept out of scope here exactly as phase 2 kept the wiring out of `coverage-threshold-gate` and `e2e-cli-smoke`.

**Pure evaluator + thin runner, mirroring the prior gates.** The risky, must-be-correct part is "is there a vulnerability at/above the configured severity, fail-closed?". That is isolated as a pure function so every branch is unit-testable without running a real audit, precisely as `evaluateCoverage` / `evaluateE2e` isolated their decisions. The runner is impure glue — read the report + threshold from env, translate the verdict into an exit code — and is exercised directly with a fixture report, no Actions runner needed. This keeps the security gate provable, not hoped.

**Signal shape is the integration contract.** `evaluateDependencyAudit` returns `signal: 'green' | 'red'` — the same `GateSignal` the release-decision module keys its gates by. That is the seam the wiring change snaps into: it will combine this slice's signal (and the secret-scan slice's) into the security signal the release-gate step passes to `decideRelease`, with no change to the core logic, which already treats gates as data. Pinning the shape now (and asserting type-compatibility in tests) is what makes the later wiring a small addition.

**Severity threshold, fail-closed on the unknown.** Failing on `high` and above is the conventional release-blocking bar: low/moderate advisories (often unfixable transitive noise) do not turn a green tree red, but a high/critical vulnerability — the kind a release must not ship — trips the gate. The bar is data (`AUDIT_FAIL_ON`) so it can be tightened to `critical` or loosened without touching code. Conversely, a missing or malformed report is treated as `red`: the safe posture for a security gate, asserted directly — an audit that never ran (or crashed before writing its report) must never read as green.

**Gate runner owns the decision, not the raw audit exit.** `pnpm audit` exits non-zero merely because *some* advisory exists, at *any* severity — too blunt to gate a release on. So the CI step tolerates the audit's own exit (`|| true`) and lets the unit-tested gate runner decide red/green against the configured severity. Enforcement lives in tested code, not in the audit tool's all-or-nothing exit — matching the batch's thesis that the gate is provable code, not a tool's default hoped to be correct.

**Structure asserted via the parser, behavior via the runner.** A unit test cannot run GitHub Actions, so the CI half is proven structurally against the parsed model the parser helper exposes: an audit step that writes a JSON report, an audit-gate step after the `test` step, and the absence of any security signal on the release path (the boundary). The decision half is proven behaviorally by exercising the evaluator and runner directly with fixture reports. The real red/green behavior runs for real on every push, which the batch wants observable.

**Trade-offs.** Splitting "produce the dependency-audit signal" (here) from "consult it in the release decision" (the `after` change) is deliberate: it keeps each slice independently provable and the release-decision spine untouched until both phase-3 signals (audit + secret scan) exist, so they are wired together once rather than the module churning twice. The cost is that the full phase proof (`security-gate.sh`) only goes green after all three changes land; this slice's own features and unit tests stand on their own in the meantime.
