# wire-security-into-release-gate

## Why

The `ci-npx-release` batch publishes `ratchet` to npm only when every quality gate is green, and that "only when green" must be a **real, unit-tested decision**, not just YAML wiring. Phase 1 shipped the pure `decideRelease` spine (`release-decision.ts`) and the thin `release-gate` runner (`release-gate.ts`), wiring `lint` and `test`. Phase 2 added `coverage` and `e2e` the same way — pure evaluators + thin runners, then a wiring change that plugged both into the spine.

Phase 3 (`security-layer-gate`) makes a **known vulnerability** or a **leaked secret** block the release. The two prior slices already PRODUCE their signals — `dependency-audit-gate` (`dependency-audit-gate.ts`) and `secret-scan-gate` (`secret-scan-gate.ts`) — each emitting the `GateSignal` shape the spine consumes and each enforced as its own red/green CI step, but **deliberately not wired** into the decision.

This change is the **wiring slice**. It joins those two security signals into a single **`security`** gate on the release-decision spine so a vulnerability or a leaked secret flips the gate to **DENY**, and ships the phase proof harness (`test/e2e/security-gate.sh`). The publish path stays `npm publish --dry-run` — this phase proves the gate, it does not ship a real release.

## What Changes

A thin, data-only wiring — the riskiest parts (the two evaluators, the two CI steps) are already proven, so this slice should be small:

- **Extend the wired-gate set, not the decision logic.** The release-decision module is intentionally data-driven: its wired-gate set is the keys of the `gates` record, not hardcoded branching. `decideRelease` needs **no change**. Wiring is: add `security` to `WIRED_GATES` in `release-gate.ts` (→ `['lint', 'test', 'coverage', 'e2e', 'security']`). The runner already maps each gate name to `GATE_<NAME>` and treats any non-`green` value (including missing) as not-green, so `security` inherits fail-closed semantics for free.
- **Fold both security steps into ONE signal in the workflow.** Phase 3's success criterion speaks of a single "security signal green before ALLOW", and the two sibling slices were written to be "combined into the single security signal the release-gate step passes to `decideRelease`". So the release-gate step in `.github/workflows/ci.yml` adds **one** `GATE_SECURITY` that is `green` only when **both** the `audit` step and the `secret-scan` step succeeded, fail-closed to `'red'` when either did not — `${{ steps.audit.outcome == 'success' && steps.secret-scan.outcome == 'success' && 'green' || 'red' }}`. This mirrors the existing per-gate `${{ steps.X.outcome == 'success' && 'green' || 'red' }}` pattern, combined across the two security steps. The dry-run publish step is unchanged.
- **Ship the phase proof harness.** Add `test/e2e/security-gate.sh` (the phase proof-of-work): build the package, then compose the already-shipped runners (`dependency-audit-gate.js`, `secret-scan-gate.js`, `release-gate.js`) over real/forced fixture reports and assert (a) a clean tree (no vulnerabilities, no findings) on `main` → both security signals green → `GATE_SECURITY` green → release gate ALLOW and the dry-run publish path is exercised; (b) a planted vulnerable dependency (an audit report with a high/critical count) makes the audit gate red → security red → release gate DENY; (c) a planted secret (a scan report with a finding) makes the secret-scan gate red → security red → release gate DENY. It composes shipped runners over forced inputs — no new decision logic.

## Design

**Wired-gate set as data.** `WIRED_GATES` in `src/core/ci/release-gate.ts` becomes `['lint', 'test', 'coverage', 'e2e', 'security']`. `decideRelease` is untouched: it already treats the wired-gate set as the keys of the `gates` record, so `security` inherits the same "ALLOW on `main` only when every wired gate is green, else DENY with a per-gate reason" semantics, and any missing/non-`green` value is fail-closed to not-green for free.

**One combined `security` signal, not two gates.** The audit and secret-scan slices each ship their own red/green CI step, but the spine consumes a single `security` gate — matching the phase's "the security signal green before ALLOW" framing and the sibling plans' stated intent to feed "the single security signal" into the decision. The workflow does the folding: `GATE_SECURITY` is `green` only when both security steps succeeded, `red` otherwise. This keeps the security layer one auditable gate on the spine while each underlying check stays an independently-red CI step. (Either security step going red also short-circuits the job via Actions' default `success()` condition, so the release path is never reached on a red security step; the `|| 'red'` fold is belt-and-braces, fail-closed if a future change relaxes the short-circuit.)

**Workflow env feeds the signal.** The release-gate step in `.github/workflows/ci.yml` adds `GATE_SECURITY: ${{ steps.audit.outcome == 'success' && steps.secret-scan.outcome == 'success' && 'green' || 'red' }}` alongside the existing `GATE_LINT` / `GATE_TEST` / `GATE_COVERAGE` / `GATE_E2E`. The `npm publish --dry-run` step is unchanged.

**Proof harness.** `test/e2e/security-gate.sh` mirrors `test/e2e/release-gate.sh`: build, then drive the shipped runners over fixture reports. It derives each security signal from its runner's exit (`green` on exit 0, fail-closed `red` otherwise), folds them into `GATE_SECURITY` exactly as the workflow does, and feeds that plus lint/test/coverage/e2e green into `release-gate.js` on `main`. Clean reports → ALLOW and the dry-run publish path is exercised (recorded by a marker); a forced high/critical audit count → audit red → security red → DENY, publish path not reached; a forced finding → secret-scan red → security red → DENY, publish path not reached. Inputs are FORCED by pointing the runners at scratch fixtures (`AUDIT_REPORT` / `SECRET_SCAN_REPORT`), not by sabotaging real source, so the harness is deterministic and side-effect-free.

**Structure vs behavior.** The wiring is proven two ways: behaviorally by exercising the `release-gate` runner directly with a `security` signal (ALLOW only when all five green; DENY on a red or missing `security`), and structurally by asserting the workflow's release-gate step now carries `GATE_SECURITY` folded from the audit + secret-scan step outcomes and the publish step is still `--dry-run`. The blackbox `security-gate.sh` is the end-to-end maintainer-facing proof on top.

**Trade-offs.** Keeping `decideRelease` untouched and changing only `WIRED_GATES` + the workflow env keeps the proven spine stable and makes the wiring auditable as data. Folding the two security checks into one `security` gate (rather than wiring `audit` and `secret` as two spine gates) keeps the spine's gate set aligned with the phase's single-"security signal" framing while leaving each underlying CI step independently red; the cost is that the spine cannot, from its signal alone, distinguish a vulnerability DENY from a secret DENY — but the per-step CI output and the harness's per-runner assertions make the cause obvious where it matters. The harness forces regressions by manipulating the gates' inputs (a high-count audit report, a finding) rather than by sabotaging real source, so it is deterministic.
