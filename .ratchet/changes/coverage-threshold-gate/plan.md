# coverage-threshold-gate

## Why

The `ci-npx-release` batch publishes `ratchet` to npm only when every quality gate is green, and its promise is that "only when green" is a **real, unit-tested decision**, not just YAML wiring. Phase 1 shipped the dry-run pipeline skeleton plus the pure, unit-tested `decideRelease` spine and the thin `release-gate` runner that bridges the workflow's environment to it (`src/core/ci/release-decision.ts`, `src/core/ci/release-gate.ts`).

Phase 2 (`coverage-and-e2e-gates`) makes **missing coverage** and **broken end-to-end behavior** block the release. That phase is sliced into three changes: this one (`coverage-threshold-gate`), `e2e-cli-smoke`, and `wire-coverage-e2e-into-release-gate` (which runs `after` both and plugs their signals into the release-decision spine).

This change is the **coverage slice**: a pure, exhaustively unit-tested evaluator that turns a measured coverage total into a `green`/`red` **gate signal** against an enforced minimum threshold, plus the CI wiring that runs coverage and goes red when it drops below the threshold. It deliberately mirrors the phase-1 split â€” a pure module + a thin runner + a minimal CI step â€” and **does not** wire the coverage signal into `decideRelease`'s wired-gate set. That wiring belongs to `wire-coverage-e2e-into-release-gate`. This slice exists to prove the "coverage >= threshold" decision in isolation so the wiring change has a trustworthy signal to feed in.

## What Changes

- **Pure coverage evaluator.** Add a shippable module under `src/core/ci/` (e.g. `src/core/ci/coverage-gate.ts`) exporting a pure `evaluateCoverage(input)` that takes a parsed coverage total (a number, 0â€“100) and a minimum `threshold` and returns an explicit `{ signal: 'green' | 'red'; coverage: number | null; threshold: number; reasons: string[] }`. `green` iff `coverage` is a real number AND `coverage >= threshold`; otherwise `red` with a precise reason (below-threshold, or "summary could not be read" when the total is missing/NaN). The returned `signal` is exactly the `GateSignal` shape (`'green' | 'red'`) the release-decision module already consumes, so the later wiring change feeds it in unchanged.
- **Threshold as data.** Define a default threshold constant (e.g. `DEFAULT_COVERAGE_THRESHOLD`) and let it be overridden via an env var (e.g. `COVERAGE_THRESHOLD`), so the value is configurable, not hardcoded at the call site. Set the default at or just below the **currently measured** total coverage so a green tree stays green and a real regression trips the gate â€” measure first, then set.
- **Summary parsing, fail-closed.** Add a small reader that extracts the total coverage percentage from the coverage tool's machine-readable summary (the v8 `json-summary` reporter writes `coverage/coverage-summary.json` with a `total` block). A missing, unreadable, or malformed summary yields `null`, which the evaluator treats as `red` â€” fail-closed, matching the spine's posture.
- **Thin coverage-gate runner.** Add a thin, impure entrypoint (e.g. `runCoverageGate(env)` in the same module, run when invoked directly Ă  la `release-gate.ts`'s `isDirectRun()`) that resolves the summary path + threshold from the environment, reads the total, calls `evaluateCoverage`, prints the outcome and any reasons, and exits `0` on `green` / non-zero on `red`. It adds **no** decision logic â€” it only adapts CI's world to the pure evaluator, exactly as `release-gate.ts` does for `decideRelease`.
- **Coverage reporter.** Extend `vitest.config.ts` coverage `reporter` to include `'json-summary'` (alongside the existing `text`/`json`/`html`) so the summary file the evaluator reads is emitted. No threshold is configured in vitest itself â€” enforcement lives in the tested evaluator/runner, keeping the gate provable rather than buried in tool config.
- **CI step.** Add a coverage step to `.github/workflows/ci.yml`, positioned **after** the existing `test` step and **before/outside** the release path: run the suite with coverage (e.g. `pnpm vitest run --coverage`) and then invoke the built coverage-gate runner (`node dist/core/ci/coverage-gate.js`) so a coverage total below the threshold exits non-zero and turns the run red. Do **not** add a `GATE_COVERAGE` signal to the release-gate step and do **not** extend `WIRED_GATES` â€” that is `wire-coverage-e2e-into-release-gate`'s job.
- **Tests** under `test/ci/` (so they run with the suite):
  - `test/ci/coverage-gate.test.ts`: `green` at and above threshold; `red` below (reason names the shortfall); fail-closed `red` on a missing/NaN total (reason names the unreadable summary); threshold override respected; the returned `signal` is assignable to the release-decision `GateSignal` type.
  - Coverage CI-step assertions (using the existing `test/ci/helpers/workflow.ts` parser): a step runs the suite with coverage collection; a coverage-gate step invokes the evaluator and sits after the `test` step; and the release path still wires only `lint`/`test` (no `GATE_COVERAGE` yet).
- Implements `features/ci/coverage-gate.feature` and `features/ci/coverage-ci-step.feature`.
- Multi-agent surface: **none**. A pure evaluator, a thin runner, vitest config, workflow YAML, and tests; no agent-specific files, skills, or commands.

## Design

**Thin vertical slice, same spine.** The phase goal is "coverage drop OR e2e failure flips the gate to DENY". The full proof (`bash test/e2e/release-gate.sh`) spans all three phase-2 changes. This change owns the smallest end-to-end coverage piece: measure coverage â†’ parse the summary â†’ decide green/red against a threshold â†’ emit a gate signal in the spine's shape, with CI going red on a below-threshold run. Joining that signal to `decideRelease` (extending `WIRED_GATES`, feeding `GATE_COVERAGE`) is the explicit `after` change â€” kept out of scope here exactly as phase 1 kept the `ci.yml` wiring out of `release-decision-module`.

**Pure evaluator + thin runner, mirroring phase 1.** The risky, must-be-correct part is "is coverage >= threshold, fail-closed?". That is isolated as a pure function so every branch is unit-testable without a coverage run, precisely as `decideRelease` isolated "only when green". The runner is impure glue â€” read summary + threshold from env, translate the verdict into an exit code â€” and is exercised directly with a fixture summary, no Actions runner needed. This keeps the coverage gate provable, not hoped.

**Signal shape is the integration contract.** `evaluateCoverage` returns `signal: 'green' | 'red'` â€” the same `GateSignal` the release-decision module keys its gates by. That is the seam the wiring change snaps into: it will read `GATE_COVERAGE` (set from this runner's outcome) and pass it through to `decideRelease` with no change to the core logic, which already treats gates as data. Pinning the shape now (and asserting type-compatibility in tests) is what makes the later wiring a one-line addition.

**Enforcement in tested code, not tool config.** Rather than set `coverage.thresholds` in `vitest.config.ts` (where the gate would be untested config the batch's thesis distrusts), the threshold is enforced by the unit-tested evaluator/runner. Vitest's role is reduced to *emitting* the `json-summary`; *deciding* on it is the tested module. This matches the batch's promise that the gate is provable code, not YAML/config hoped to be correct.

**Threshold set from reality, fail-closed on the unknown.** The default threshold is anchored at or just below the current measured coverage so a green tree stays green and a regression trips it â€” a threshold above current coverage would make CI red on the very push that introduces the gate, which the phase (still dry-run, observable on a real push) does not want. Conversely, a missing or malformed summary is treated as `red`: the safe posture for a release gate, asserted directly.

**Structure asserted via the parser, behavior via the runner.** A unit test cannot run GitHub Actions, so the CI half is proven structurally against the parsed model the prior phase's helper exposes: a coverage-collecting test step, a coverage-gate step after it, and the absence of any `GATE_COVERAGE` on the release path (the boundary). The decision half is proven behaviorally by exercising the evaluator and runner directly. The real red/green behavior runs for real on every push, which the batch wants observable.

**Trade-offs.** Splitting "produce the coverage signal" (here) from "consult it in the release decision" (the `after` change) is deliberate: it keeps each slice independently provable and the release-decision spine untouched until both phase-2 signals (coverage + e2e) exist, so they are wired together once rather than the module churning twice. The cost is that the full phase proof (`release-gate.sh`) only goes green after all three changes land; this slice's own features and unit tests stand on their own in the meantime.
