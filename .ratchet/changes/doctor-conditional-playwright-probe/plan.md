# doctor-conditional-playwright-probe

## Why

`ratchet eval` now supports a `kind: web` binding that drives Playwright specs (`web-binding-schema`, `web-lifecycle-harness`, `web-deterministic-fold`, `web-failure-evidence`), but nothing yet tells a user whether Playwright is actually installed before they hit a confusing failure mid-run. `ratchet doctor` must surface that check — but only for users who actually use web bindings, since Playwright stays an opt-in, user-owned dependency the same way Docker is opt-in for the `docker` locus.

## What Changes

- A new, project-scoped helper resolves whether any `kind: web` binding exists among the eval bindings resolved from `.ratchet/evals/specs/` (`loadEvalSpecs`).
- A new `checkPlaywright` doctor check (`src/core/doctor/checks/playwright.ts`) detects whether the Playwright CLI is usable (`npx --no-install playwright --version`) and reports `pass` (with detected version) or `info` (with install remedy) — `optional` severity, mirroring `checkDocker`'s never-blocks pattern.
- `runDoctorChecks` (`src/core/doctor/index.ts`) appends the Playwright check to the report **only when** a `kind: web` binding is present in scope; the check is entirely absent from `report.checks` otherwise (not merely hidden/skipped-status — literally not in the array), and absent from `--json` output too.
- Implements the Given/When/Then in `features/doctor-playwright-probe/scope-gating.feature` and `features/doctor-playwright-probe/detection.feature`.
- Documents the conditional check in `docs/commands/doctor.md` and `docs/commands/eval.md` (replacing the existing "still deferred" placeholder note), and adds Playwright as a conditional requirement row in `README.md`.
- No breaking changes: `runDoctorChecks()` keeps working with zero arguments (project root defaults to the resolved planning-home root), and every existing check/report shape is untouched.

## Design

- **Scope detection reuses the existing binding resolver, not a new schema.** `hasWebBindingInScope(projectRoot)` (new, `src/core/doctor/web-scope.ts`) calls `loadEvalSpecs(projectRoot)` from `src/core/eval/index.ts` and returns `true` iff any `ResolvedBinding` in `.bindings` has `binding.kind === 'web'`. This is the same source of truth `eval set`/`eval run` already use to resolve bindings, so "in scope" here means "resolved from the project's eval specs" — there is no second, divergent notion of scope. A project with no `.ratchet/evals/specs/` directory, or only `deterministic`/`llm-judge` bindings, or spec files that fail to parse/validate, all resolve to `false` (the resolver already tolerates missing dirs and warns-and-skips invalid entries; doctor does not need its own error handling for that).
- **Conditional inclusion lives in the orchestrator, not the check.** `runDoctorChecks(deps, projectRoot)` gains an optional second parameter (defaulting to `resolveCurrentPlanningHomeSync().root`, the same helper `src/commands/eval/shared.ts` uses) purely to make `hasWebBindingInScope` injectable/testable. It builds the checks array as `[checkAgents, checkRuntime, checkDocker]` and pushes `checkPlaywright(deps)` only when `hasWebBindingInScope(projectRoot)` is true. `checkPlaywright` itself stays a pure, unconditional function (same shape as `checkDocker`) — the *scope* decision is the orchestrator's job, keeping each check a simple `(deps) => DoctorCheck` function.
- **Detection command mirrors the harness's own invocation.** The web lifecycle harness already runs Playwright specs via `npx playwright test ...` (`src/core/eval/web-lifecycle.ts`). The doctor probe reuses `npx` for the same reason — Playwright is inherently an npm-ecosystem tool once a project opts into a `kind: web` binding (the schema requires a Playwright spec path), so this is detecting the actual declared dependency, not assuming a package manager for the user's general project (`generalizable-defaults`). `--no-install` is required so the probe never triggers npx's implicit network install when Playwright isn't present — a doctor check must be fast and side-effect-free.
- **Severity is `optional`, status is `pass`/`info`, never `fail`.** Exactly like `checkDocker`: Playwright's absence never fails `ratchet doctor` or gates its exit code, since it is only needed when the user has opted into web bindings, and even then only for the `eval run`/`eval record` path — never for the core batch lifecycle `doctor` otherwise gates.
- **No new `DoctorCheck` shape or rendering logic.** The check slots into the existing `checks[]` array/`isReportOk`/`renderReport`/`serializeReport` machinery untouched — `render.ts` already handles arbitrary `id`/`status`/`severity` generically (see how `docker` already renders with an `ℹ` info line and a `→` remedy), so no changes are needed there.
- **Documentation:** per the `documentation` standard, `docs/commands/doctor.md` gets a fourth "Checks" subsection ("Playwright CLI (`playwright`) — optional, conditional") describing when it appears and its pass/info conditions, and the "Three checks run in a fixed order" line is updated to note the conditional fourth. `docs/commands/eval.md`'s "Web binding" section already has a placeholder sentence ("The conditional `ratchet doctor` Playwright probe is still deferred to a later change...") — this change replaces that sentence with the real, implemented behavior and a cross-reference to `doctor.md`. `README.md`'s dependency table gets a new "Playwright" row scoped to "Only when a `kind: web` eval binding is in scope", matching the existing Docker row's "Optional" framing.
- **Testing:** per the `testing` standard, `hasWebBindingInScope` and the conditional `runDoctorChecks` wiring are integration-tested over a tmpdir fixture repo (writing real `.ratchet/evals/specs/*.yaml`), since they touch the filesystem through `loadEvalSpecs`; `checkPlaywright` itself is unit-tested with the existing `FakeDeps` pattern from `test/core/doctor/doctor.test.ts` (no fs, deps injected) since it is pure. New test files are named with a `web-` prefix (`test/core/doctor/web-scope-gating.test.ts`, `test/core/doctor/web-playwright-detection.test.ts`) so the phase's proof-of-work (`pnpm vitest run web`) picks them up, matching the existing `test/core/eval/web-lifecycle.test.ts` naming convention. Each test file's header names the `.feature` it implements.
