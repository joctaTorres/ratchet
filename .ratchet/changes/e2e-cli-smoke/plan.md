# e2e-cli-smoke

## Why

The `ci-npx-release` batch publishes `ratchet` to npm only when every quality gate is green, and its promise is that "only when green" is a **real, unit-tested decision**, not just YAML wiring. Phase 1 shipped the dry-run pipeline skeleton plus the pure, unit-tested `decideRelease` spine and the thin `release-gate` runner that bridges the workflow's environment to it (`src/core/ci/release-decision.ts`, `src/core/ci/release-gate.ts`).

Phase 2 (`coverage-and-e2e-gates`) makes **missing coverage** and **broken end-to-end behavior** block the release. That phase is sliced into three changes: `coverage-threshold-gate` (already done â€” the coverage signal), this one (`e2e-cli-smoke`), and `wire-coverage-e2e-into-release-gate` (which runs `after` both and plugs their signals into the release-decision spine).

This change is the **e2e slice**: an end-to-end smoke that drives the **built** CLI the way a user running `npx ratchet` would â€” spawning `bin/ratchet.js` as a subprocess and asserting on its exit code and stdout â€” plus a pure, unit-tested evaluator that turns the smoke's machine-readable result into a green/red **gate signal**, and the CI wiring that runs the smoke and goes red when it fails. It deliberately mirrors the coverage slice â€” a real signal-producer + a pure evaluator + a thin runner + a minimal CI step â€” and **does not** wire the e2e signal into `decideRelease`'s wired-gate set. That wiring belongs to `wire-coverage-e2e-into-release-gate`. This slice exists to prove the "the built CLI works end to end" decision in isolation so the wiring change has a trustworthy signal to feed in.

## What Changes

- **End-to-end CLI smoke.** Add a shell smoke at `test/e2e/cli-smoke.sh` that builds the package (`pnpm build`) and then drives the built CLI **as a subprocess** â€” `node bin/ratchet.js <args>`, the same path `npx ratchet` resolves through the package `bin` â€” across a few user-facing checks: `--version` exits 0 and prints the package version; `--help` exits 0 and lists the available commands; at least one read-only subcommand (e.g. `status` or `list`) runs end to end in a scratch directory and exits cleanly. It asserts on the process exit code and stdout only â€” never importing internals â€” so a build/packaging/entrypoint regression that unit tests miss is caught. The script lives alongside the existing `test/e2e/*.sh` smokes and follows their conventions (`set -euo pipefail`, a temp workdir, clear per-check logging).
- **Machine-readable result, fail-closed.** The smoke writes a machine-readable result file (e.g. `test/e2e/.results/cli-smoke.json`) capturing each check's name + pass/fail and an overall `ok`, so the evaluator has real data to act on. A missing, unreadable, or malformed result yields `null`, which the evaluator treats as `red` â€” fail-closed, matching the spine's posture.
- **Pure e2e evaluator.** Add a shippable module under `src/core/ci/` (e.g. `src/core/ci/e2e-gate.ts`) exporting a pure `evaluateE2e(input)` that takes the parsed smoke result and returns an explicit `{ signal: 'green' | 'red'; ok: boolean | null; failures: string[]; reasons: string[] }`. `green` iff the result parsed AND `ok` is true AND no check failed; otherwise `red` with a precise reason (names the failing check(s), or "e2e result could not be read" when the result is missing/unparseable). The returned `signal` is exactly the `GateSignal` shape (`'green' | 'red'`) the release-decision module already consumes, so the later wiring change feeds it in unchanged.
- **Thin e2e-gate runner.** Add a thin, impure entrypoint (e.g. `runE2eGate(env)` in the same module, run when invoked directly Ă  la `release-gate.ts`'s `isDirectRun()`) that resolves the result path from the environment, reads + parses it, calls `evaluateE2e`, prints the outcome and any reasons, and exits `0` on `green` / non-zero on `red`. It adds **no** decision logic â€” it only adapts CI's world to the pure evaluator, exactly as `release-gate.ts` does for `decideRelease`.
- **CI step.** Add an e2e step to `.github/workflows/ci.yml`, positioned **after** the existing `test`/`coverage` steps and **before/outside** the release path: build the package and run the smoke (`bash test/e2e/cli-smoke.sh`), then invoke the built e2e-gate runner (`node dist/core/ci/e2e-gate.js`) so a failed smoke check exits non-zero and turns the run red. Do **not** add a `GATE_E2E` signal to the release-gate step and do **not** extend `WIRED_GATES` â€” that is `wire-coverage-e2e-into-release-gate`'s job.
- **Tests** under `test/ci/` (so they run with the suite):
  - `test/ci/e2e-gate.test.ts`: `green` when the parsed result is `ok` with all checks passed (no reasons); `red` when any check failed (reason names the failing check); fail-closed `red` on a missing/unparseable result (reason names the unreadable result); and that the returned `signal` is assignable to the release-decision `GateSignal` type.
  - E2e CI-step assertions (using the existing `test/ci/helpers/workflow.ts` parser): a step builds the package; a step runs the e2e smoke against the built binary; an e2e-gate step invokes the evaluator and sits after the `test` step; and the release path still wires only `lint`/`test` (no `GATE_E2E` yet).
- Implements `features/e2e/cli-smoke.feature` and `features/ci/e2e-ci-step.feature`.
- This slice owns `test/e2e/cli-smoke.sh` (the e2e half of the phase). The phase-wide proof `test/e2e/release-gate.sh` â€” which asserts a passing run keeps the gate ALLOW and that a forced coverage-below-threshold OR a failing e2e flips the decision to DENY â€” is assembled by the `after` change `wire-coverage-e2e-into-release-gate`, once both phase-2 signals exist and are wired into `decideRelease`.
- Multi-agent surface: **none**. A shell smoke, a pure evaluator, a thin runner, workflow YAML, and tests; no agent-specific files, skills, or commands.

## Design

**Thin vertical slice, same spine.** The phase goal is "coverage drop OR e2e failure flips the gate to DENY". The full proof (`bash test/e2e/release-gate.sh`) spans all three phase-2 changes. This change owns the smallest end-to-end e2e piece: drive the built CLI as a user â†’ record a machine-readable result â†’ decide green/red against it â†’ emit a gate signal in the spine's shape, with CI going red on a failed smoke. Joining that signal to `decideRelease` (extending `WIRED_GATES`, feeding `GATE_E2E`) is the explicit `after` change â€” kept out of scope here exactly as the coverage slice kept its wiring out, and as phase 1 kept the `ci.yml` wiring out of `release-decision-module`.

**Drive the built artifact, not the source.** The risky thing an e2e gate must catch is a regression that only appears once the package is *built and run as a binary* â€” a broken `bin` shebang, a missing `dist/` entry, an ESM/resolution fault, a packaging gap. So the smoke spawns `node bin/ratchet.js` as a child process against a fresh `pnpm build` and asserts purely on exit code + stdout, the same surface `npx ratchet` exposes to a user. Importing CLI internals would defeat the point and merely duplicate unit tests; subprocess execution is the commitment.

**Pure evaluator + thin runner, mirroring the coverage slice.** The must-be-correct part is "did every check pass, fail-closed?". That is isolated as a pure function of the parsed result so every branch is unit-testable without running the smoke, precisely as `evaluateCoverage` isolated "coverage >= threshold" and `decideRelease` isolated "only when green". The runner is impure glue â€” read the result path from env, translate the verdict into an exit code â€” and is exercised directly with a fixture result, no Actions runner needed. This keeps the e2e gate provable, not hoped.

**Signal shape is the integration contract.** `evaluateE2e` returns `signal: 'green' | 'red'` â€” the same `GateSignal` the release-decision module keys its gates by. That is the seam the wiring change snaps into: it will read `GATE_E2E` (set from this runner's outcome) and pass it through to `decideRelease` with no change to the core logic, which already treats gates as data. Pinning the shape now (and asserting type-compatibility in tests) is what makes the later wiring a one-line addition â€” symmetric with the `coverage` signal the prior slice pinned.

**Fail-closed on the unknown.** A missing or malformed result is treated as `red`: the safe posture for a release gate (a smoke that never ran, or crashed before writing its result, must not read as green), asserted directly â€” mirroring the coverage slice's treatment of an unreadable summary.

**Structure asserted via the parser, behavior via the runner and smoke.** A unit test cannot run GitHub Actions, so the CI half is proven structurally against the parsed model the prior phase's helper exposes: a build step, an e2e-smoke step, an e2e-gate step after the `test` step, and the absence of any `GATE_E2E` on the release path (the boundary). The decision half is proven behaviorally by exercising the evaluator and runner directly against fixture results; the smoke itself runs for real against the built CLI in CI and locally, which the batch wants observable.

**Trade-offs.** Splitting "produce the e2e signal" (here) from "consult it in the release decision" (the `after` change) is deliberate: it keeps each slice independently provable and the release-decision spine untouched until both phase-2 signals (coverage + e2e) exist, so they are wired together once rather than the module churning twice. The cost is that the full phase proof (`release-gate.sh`) only goes green after the wiring change lands; this slice's own features, unit tests, and the standalone `cli-smoke.sh` stand on their own in the meantime. Keeping the smoke read-only (version/help + a non-mutating subcommand in a scratch dir) trades exhaustive command coverage for a fast, deterministic, side-effect-free signal appropriate to a release gate.
