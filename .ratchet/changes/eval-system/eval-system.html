ratchet eval

Turn the project's .feature files into a scored, reproducible regression suite — judged against fixed fixtures, never the live repo.

The idea in one paragraph

Ratchet already describes its behavior as Gherkin .feature files. The eval system reads those files and makes one eval case per scenario. Each case is bound to a fixture — a small, checked-in codebase with a known-good state — and judged by the batch engine. Same input every time, so a scenario that passes today can't quietly break tomorrow. The CLI does the deterministic work (enumerate, run, score, diff); the engine does the judging.

Two ways to judge a case

check — deterministic

Run a bash command in the fixture and check its output: exit-zero, contains:…, or regex:…. No agent. Always the same answer.

agent — spawned judge

For prose-y scenarios, spawn a fresh coding agent in the fixture, judge it against the binding's success criteria, capture pass/fail + reason.

Both run inside a throwaway copy of the fixture — never the live working tree. Pick the mode with --judge auto|check|agent.

The loop

01setlist cases from .feature files
→
02runengine judges each bound case
→
03reportscorecard + baseline diff
→
04baselinepromote a clean run

Commands

ratchet eval setList eval cases from .feature files. Scope with --changes, --change <name>, --path. Archive excluded.
ratchet eval runJudge every bound case through the engine and persist the run. --judge auto|check|agent.
ratchet eval recordManually override one verdict (for cases with no fixture). fail needs evidence.
ratchet eval reportScorecard, failing cases with evidence, and the baseline regression diff.
ratchet eval baselinePromote a run to the baseline future runs are compared against.

Binding a case

Cases come from .feature files; an eval-spec says how to judge each one. Unbound cases report unjudged — they never count as a pass.

# .ratchet/evals/specs/status.yaml
features/cli-workflow/status#status-as-json:
  fixture: status-ok          # .ratchet/evals/fixtures/status-ok/
  kind:    check
  setup:   pnpm install          # optional: runs ONCE into a cached copy
  check:
    run:   ratchet status --change add-login --json
    pass:  contains:applyRequires

features/agent-workflows/verify#findings-graded-by-severity:
  fixture:    verify-sample
  kind:       agent
  success:    each finding is graded CRITICAL, WARNING or SUGGESTION
  agentVotes: 3                # N-of-M repeat votes; majority wins

Verdicts & baseline

pass fail unjudged regression
  • Regression — passed in the baseline, fails now. The run's verdict is failing while any regression exists.
  • New / retired — a case only in this run, or only in the baseline. Diffed, not failed.
  • Unjudged — no binding yet. Keeps the run incomplete; never a pass.

Keeping the judge honest

The spawned-agent judge is the one place reproducibility could leak, so it is constrained — judge noise can never manufacture a baseline regression:

  • Fail closed on uncertainty — a verdict with no concrete evidence is not a pass; the missing evidence is named in the reason.
  • N-of-M votes — agentVotes (default 1) re-runs the judge and takes the majority. When the votes disagree, the case is recorded unjudged with the disagreement noted — never silently fail.
  • Deterministic checks preferred — a check needs none of this, so it stays the first choice; the agent judge is the fallback.

Fixtures are materialized into a throwaway working copy per case. When a fixture needs bootstrapping, its binding's one-time setup runs once into a copy cached by fixture+setup, and every case bound to that fixture reuses it — so a serial run isn't dominated by repeated install cost.

Why it's built this way

  • Reproducible — fixtures are fixed inputs, so verdicts don't drift with the live repo.
  • No new judge — reuses the batch engine's existing check + spawn seams.
  • Ratchet-forward — the baseline diff is the guardrail: green scenarios stay green.
  • Tool-agnostic — the /rct:eval skill is generated for every supported agent.
