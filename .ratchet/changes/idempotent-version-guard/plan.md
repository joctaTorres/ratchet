# idempotent-version-guard

## Why

The `ci-npx-release` batch publishes `ratchet` to npm only on a green build of `main`. The prior slice, `gated-publish-job`, promoted the publish into its own `publish` job gated on `release_allowed == 'true'` from the unit-tested `decideRelease`. That job still runs `npm publish --dry-run`.

The phase requires the release to be **idempotent**: a re-run of an already-published version must not error the pipeline. Today nothing enforces that. `npm publish` of a version already on the registry fails non-zero (E409), so a green re-run of an unchanged version would turn RED once the publish is real. This slice closes that gap before `real-npm-publish` makes the publish real.

This **idempotent-version-guard** slice adds a guard that, on the permitted publish path, decides PUBLISH (new version) vs SKIP (already published) and makes SKIP a deliberate green no-op. It stays `npm publish --dry-run` and forces the already-published set via environment; the real registry query and real publish are the last `after` change.

## What Changes

A thin end-to-end slice through the whole stack — pure decision → runner output → gated publish step → blackbox idempotency proof — that adds the version guard without touching `decideRelease` and without performing a real release:

- **A pure version-decision module.** Add `src/core/ci/version-decision.ts` exporting `decidePublishVersion({ version, publishedVersions })` → `{ shouldPublish: boolean, outcome: 'PUBLISH' | 'SKIP', reasons: string[] }`. It is pure (no I/O, no registry, no clock), mirroring `release-decision.ts`: `SKIP` when `version` is in `publishedVersions` (already published — idempotent no-op), `PUBLISH` otherwise. This is the new, exhaustively unit-testable spine of idempotency.

- **A thin impure runner.** Add `src/core/ci/version-guard.ts` (mirroring `release-gate.ts`): it reads the local version (from `package.json`, overridable via env for tests) and the set of already-published versions (FORCED via an env var such as `PUBLISHED_VERSIONS`, comma-separated — the same forcing posture the `GATE_*` signals use), calls `decidePublishVersion`, prints the outcome, writes `should_publish=true|false` to the file named by `GITHUB_OUTPUT`, and **always exits 0** — a SKIP must never error the pipeline. The pure value-returning function stays separately testable from the impure `GITHUB_OUTPUT`/`process.exit` path, exactly as `runReleaseGate` / `writeReleaseAllowedOutput` are split.

- **Wire the guard into the publish job.** In `.github/workflows/ci.yml`, inside the existing gated `publish` job, add a version-guard step (with an `id`) that runs the runner and exposes a `should_publish` step output, then condition the existing publish step on `steps.<guard>.outputs.should_publish == 'true'`. An already-published version skips the publish step while the `publish` job — and the whole pipeline — stays green. The publish step stays `npm publish --dry-run`; no npm token/secret or provenance permission is added here.

- **Unit + structural tests.** Prove (a) the pure module: `PUBLISH` with empty reasons for a new version, `SKIP` with an explanatory reason for an already-published one (including edge cases — empty published set, version present among several); (b) the runner: writes `should_publish=true` to a scratch `GITHUB_OUTPUT` for a new version and `should_publish=false` for an already-published one, and exits 0 in BOTH cases; (c) structurally (extending `test/ci/helpers/workflow.ts` usage / a new `test/ci/idempotent-version-guard.test.ts`): the `publish` job has a version-guard step exposing `should_publish` and the publish step is conditioned on `should_publish == 'true'` and is still `npm publish --dry-run`.

- **Thicken the phase proof harness.** Extend `test/e2e/npx-publish.sh` (the phase-4 proof-of-work) with idempotency cases layered on the ALLOW path: with the gate at ALLOW on `main`, run the version-guard runner over a forced `PUBLISHED_VERSIONS` set and assert — a NEW version → `should_publish=true` → dry-run publish path exercised → pipeline exits 0; an ALREADY-PUBLISHED version → `should_publish=false` → publish path NOT reached → pipeline STILL exits 0 (idempotent re-run does not error). The existing gating cases (red gate / non-main → no publish) are preserved.

## Design

**Idempotency as a pure decision, sourced impurely.** The "is this version already published?" question is split, exactly as the release decision was: a pure `decidePublishVersion` decides SKIP/PUBLISH from data, and a thin runner gathers that data. In this slice the data (the published-version set) is FORCED via environment so the proof is deterministic and offline; in `real-npm-publish` the runner swaps the env source for a real `npm view ratchet versions` query — no change to the pure decision. This is the same seam `gated-publish-job` used when it forced `GATE_*` and left the real publish to a later change.

**SKIP is success, not failure.** The crux of idempotency is that an already-published version must keep the pipeline GREEN. So the runner exits 0 for BOTH outcomes; the publish vs no-publish branch is carried entirely by the `should_publish` step output that gates the publish step — never by an exit code. This is the inverse of `release-gate`'s exit-code semantics (DENY is a failure there); here SKIP is a deliberate, healthy no-op, and the design makes that explicit.

**Two independent guards compose, both fail-safe.** The publish path now passes two gates in series: the release gate (`release_allowed == 'true'`, governs whether the `publish` job runs at all) and the version guard (`should_publish == 'true'`, governs whether it actually publishes). The first is fail-CLOSED (missing/non-`true` never publishes). The second is fail-SAFE toward green (a SKIP never errors) but still fail-CLOSED toward publishing (only the literal `true` publishes; a missing/unknown signal does not publish, and does not error). Together: nothing publishes unless explicitly permitted AND explicitly new, and a re-run of an old version is a clean pass.

**Structure vs behavior, proven two ways.** Behaviorally, the pure module and the runner are exercised directly (the runner over a scratch `GITHUB_OUTPUT`, asserting both the written value and exit 0). Structurally, the parsed-workflow model asserts the guard step's `should_publish` output and the publish step's `if` condition by substring match (robust to cosmetic renames), as the sibling tests already do. The blackbox `npx-publish.sh` is the maintainer-facing end-to-end proof on top.

**Scope discipline.** This slice ships the version guard and its publish-step gating and NOTHING more: the published set is forced via env (no real `npm view`), the publish stays `npm publish --dry-run` (no token, no `id-token`/provenance permission), and there is no `npx ratchet --version` assertion against a published CLI. Those are the explicit final `real-npm-publish` change. `decideRelease`, `WIRED_GATES`, and the `release_allowed` plumbing from `gated-publish-job` are untouched.

**Trade-offs.** Forcing the published-version set via env (rather than querying the registry) means this slice does not exercise the literal `npm view` call — accepted, because it keeps the idempotency PROOF deterministic and offline, and the swap to a real query is a one-line source change covered by `real-npm-publish`. Reading the local version from `package.json` couples the runner to the manifest, but that is the single source of truth the real publish uses too. Exiting 0 on SKIP means the publish step's own skip is the only signal that nothing shipped — surfaced via the printed outcome and the `should_publish` output, and asserted by the harness, so the no-op is observable rather than silent.
