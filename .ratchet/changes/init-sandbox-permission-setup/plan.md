# init-sandbox-permission-setup

## Why

`ratchet batch apply` lazily prompts for an agent sandbox permission posture on first run, which means developers discover the concept mid-batch instead of at setup time. Offering the same project-level permission config during `ratchet init` — right after they pick their tools — makes the security posture a deliberate setup decision and lets headless batch runs start with an explicit policy.

## What Changes

- `ratchet init` (interactive mode) adds a new step **after** the "Select tools to set up" selection: a yes/no offer to set up a project-level permission config for agent sandbox execution.
- Accepting prompts for a permission posture and writes it to the project config at `.ratchet/config.yaml` (`batch.permissions`), reusing the ReX permission flow that `ratchet batch apply` uses.
- The offer is **skipped** when a project-level permission config already exists, and skipped entirely in non-interactive mode.
- Declining writes nothing and lets init continue normally.
- Implements `features/init/sandbox-permission-setup.feature`.
- Multi-agent surface: **none agent-specific**. The permission config is a tool-agnostic posture (`PermissionsPolicy`) that governs every supported coding agent; no per-agent files, skills, or commands are generated by this change.

## Design

**Reuse, don't reinvent.** The permission machinery already exists in `src/core/batch/`:
- `PermissionsPolicy` / posture values and descriptions (`permissions-policy.ts`).
- `setProjectBatchPermissions(projectRoot, policy)` (`config.ts`) writes `batch.permissions` to `.ratchet/config.yaml`.
- `first-run-setup.ts` already has a posture-selection prompt (`defaultPrompts.selectPosture`).

**Init-specific gate (key difference from batch first-run).** The batch first-run gate (`hasPermissionConfig`) skips when *either* user *or* project config exists. This change is scoped to **project level** per the requirement: the offer is skipped only when a **project-level** config exists. Detection therefore checks the project scope directly via `readProjectConfig(projectRoot)?.batch?.permissions !== undefined` rather than calling `hasPermissionConfig`. (A user-level posture still applies at runtime through normal scope merging; init just doesn't suppress the project-level offer because of it.)

**Where it hooks in.** In `src/core/init.ts` `execute()`, after `getSelectedTools(...)` returns (around line 134) and before `createDirectoryStructure`. A new private method `offerSandboxPermissionSetup(projectPath)`:
1. Returns early if not interactive (mirror init's existing interactivity check / welcome-screen gate).
2. Returns early if project-level permission config already exists.
3. Shows a `confirm` prompt (`@inquirer/prompts`) explaining the config governs what spawned coding agents may do without approval.
4. On accept: run posture selection (reuse `selectPosture`), build `{ posture }`, call `setProjectBatchPermissions(projectPath, policy)`, print a confirmation line.
5. On decline: return without writing.

Scope is always **project** (no user-vs-project scope prompt, unlike batch first-run) because the requirement is explicitly project-level.

**Trade-offs.** Reusing `setProjectBatchPermissions` and `selectPosture` keeps a single source of truth for posture semantics. The offer must be non-fatal and skippable so init never blocks on it. Since `.ratchet/config.yaml` is created later in the flow, confirm `setProjectBatchPermissions` either creates or merges into the project config correctly when called at this point (it operates on `projectRoot`, independent of init's directory-creation step — verify ordering in implementation).
