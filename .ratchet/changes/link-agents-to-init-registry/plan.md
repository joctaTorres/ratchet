# link-agents-to-init-registry

## Why

The batch coding-agent registry (`AGENT_BINARIES` in `src/core/batch/engine/agent.ts`)
is currently derived from `BUILTIN_ADAPTERS`, an independent list, so init's tool
registry (`AI_TOOLS`) and the set of spawnable coding agents can silently drift. We
want init (`AI_TOOLS`) to be the single source of truth for which coding agents exist:
agents must be a subset of init tools, and adding an init agent should automatically
make doctor probe it. Gemini is currently a batch adapter but not an init tool, which
already violates "agents ⊆ init".

## What Changes

- Add an optional `agentBinary?: string` field to `AIToolOption` (`src/core/config.ts`),
  marking which init tools are spawnable coding agents and naming the binary doctor
  probes / the engine spawns.
- Mark the agent tools: claude→`claude`, codex→`codex`, cursor→`cursor-agent`,
  gemini→`gemini`. github-copilot and opencode get NO `agentBinary` (explicitly
  non-agents).
- Add **Gemini** as a first-class init tool: `AI_TOOLS` entry (`skillsDir: '.gemini'`,
  `agentBinary: 'gemini'`), a `ToolCommandAdapter` registered in
  `src/core/command-generation/registry.ts` rendering into `.gemini`, and skills/commands
  generated by `ratchet init` per the `multi-agent-support` standard.
- Derive `AGENT_BINARIES` from the `agentBinary`-marked `AI_TOOLS` entries instead of
  from `BUILTIN_ADAPTERS`. `BUILTIN_ADAPTERS` keeps providing the spawn argv.
- Add a mandatory drift-guard test and supporting tests (see feature files
  `agent-init-link/*.feature`).
- Update now-stale expectations: command-adapter registry count (5→6), adapter list
  tests, doctor `AGENT_BINARIES` block, and any tool-enumeration tests that now
  include gemini.

Implements:
- `features/agent-init-link/derived-agent-registry.feature`
- `features/agent-init-link/registry-drift-guard.feature`
- `features/agent-init-link/gemini-init-tool.feature`
- `features/agent-init-link/doctor-preserved.feature`

## Design

**Single source of truth.** `AI_TOOLS` gains `agentBinary?: string`. An init tool is a
coding agent iff it declares `agentBinary`. This makes "which agents exist" a property
of the init registry, not a parallel list.

**Derivation direction.** `AGENT_BINARIES` is built by filtering `AI_TOOLS` to entries
with `agentBinary` and mapping `value → agentBinary`. `agent.ts` imports `AI_TOOLS`
from `config.ts`; `config.ts` does not import `agent.ts`, so there is no import cycle
(verified with madge). `BUILTIN_ADAPTERS` stays the source of spawn argv. To avoid
duplicating the binary string, the drift-guard test asserts each adapter's resolved
spawn command equals its `agentBinary` rather than coupling the two definitions; if
that proves clean we keep them independent and let the guard enforce agreement.

**Doctor.** `checkAgents` already iterates `AGENT_BINARIES`, so once `AGENT_BINARIES`
is derived from init it transparently probes every init agent (including gemini) with
no edit to the check. Its required-fail-when-none behavior is unchanged and re-verified.

**Multi-agent surface for Gemini (per the standard).** Gemini gets the full treatment:
- registry entry in `src/core/config.ts`
- `geminiAdapter` in `src/core/command-generation/adapters/gemini.ts`, exported from
  `adapters/index.ts`, registered in `registry.ts`
- generated outputs under `.gemini/` for every workflow, e.g.
  `.gemini/skills/ratchet-<workflow>/SKILL.md` and `.gemini/commands/rct-<id>.md`,
  mirroring the cursor/opencode file conventions.
No shared template logic changes — generation iterates the registry.

**Trade-offs.** Keeping `BUILTIN_ADAPTERS` as the argv source (rather than folding
binary names into `AI_TOOLS` only) preserves the existing engine seam and stream-json
capability flags; the guard test removes the drift risk that motivated the change.
