# Link standards to features

## Why

The standards library (from the `add-standards` change) lets propose and verify
reason over standards, but nothing records *which* change follows *which* standard,
or *which* shipped features satisfy a standard. Without a durable link, the
permanent spec can't answer "what enforces our security standard?" and standards
can't be traced to the behavior that implements them. This adds a hard, validated
link in both directions, materialized into the feature store at archive time.

## What Changes

- Give every standard a unique **`tag`** (frontmatter in `.ratchet/standards/<file>.md`),
  validated unique across the library; the tag falls back to the file name when
  absent. The standards loader (`src/core/standards.ts`) gains frontmatter parsing
  and a `tag` field on `StandardDoc`.
- Let a change **declare the standards it follows**: `standards: [<tag>…]` in
  `.ratchet/changes/<name>/.ratchet.yaml` (extend `ChangeMetadataSchema`). **Propose**
  records the chosen tags; **validate** errors on unknown tags; **verify** scopes its
  standards check to the declared set.
- On **archive**, materialize the links into the permanent store (change-level
  granularity, per the design decision):
  - **Forward link** → a per-capability sidecar `.ratchet/features/<capability>/.ratchet.yaml`
    mapping each `.feature` file to the change's standard tags.
  - **Reverse link** → a generated `## Implemented by` block in each
    `.ratchet/standards/<tag>.md`, regenerated by scanning the sidecars so it never
    goes stale.
- Implements the behavior in
  `features/standard-links/standard-tags.feature`,
  `features/standard-links/declare-standards-on-change.feature`, and
  `features/standard-links/materialize-links-on-archive.feature`.
- **Depends on** the `add-standards` change (PR #3, branch `feat/standards-library`),
  which introduces `.ratchet/standards/` and the loader this change extends.

## Design

**Change-level declaration, accepted tradeoff.** The forward link is declared once
per change (`standards: [tags]`), not per feature file. This is simpler to author but
coarse: every feature the change produces is attributed to every declared standard.
We chose this deliberately over feature-level Gherkin `@standard:` tags; the
imprecision is acceptable and the model stays simple.

**One source of truth; the reverse link is derived.** The authored truth is two
things: each standard's `tag`, and each change's `standards` list. Everything in the
store is *materialized* from those at archive — archive is the single writer. The
reverse link (standard → implementing features) is a pure projection of the forward
sidecars, so it is **regenerated, never hand-edited or blindly appended**. Rename or
tombstone a feature and its entry drops out on the next archive; the standard's list
can't rot. This is why "updated on archive" is safe.

**Why a per-capability sidecar (the store has no per-feature dir).** The feature store
is a flat tree of `.feature` files keyed by path; `applyFeatures` copies them
whole-file by store-relative path (features-apply.ts). There is no per-feature
directory, so the original sketch's `.ratchet/features/<feature dir>/.ratchet.yaml` has
no home at file granularity. The closest faithful grouping is the **capability dir**
(the first path segment). So the forward link lands at
`.ratchet/features/<capability>/.ratchet.yaml`, keyed by feature file name. Keeping the
forward link *in the store* (not only in the archived change) is what lets the reverse
link be regenerated purely by scanning the store, independent of archived changes.

**Tag identity vs file name.** The `tag` is the stable key; the file name is incidental
(a standard can be renamed without breaking references). When no `tag` is declared, the
loader derives it from the file name so existing single-file standards keep working.
Uniqueness is enforced at validation time, surfacing duplicates as errors rather than
silently picking one.

**Archive flow integration.** `ArchiveCommand` already calls `applyFeatures` to copy
features into the store. Standard-link materialization runs in the same phase, reusing
the `FeatureUpdate[]` (which carries `rel`, `capability`, and the add/overwrite/delete
classification) plus the change's declared tags. Deletions (tombstones) must drop
sidecar entries so the regenerated reverse blocks stay correct.

**Validation surface.** `validate` gains two checks: standard tag uniqueness across the
library, and that every tag referenced by a change's `standards` list exists. Both are
ERROR-level (a dangling reference is a real defect). Archive already validates before
copying, so a bad link is caught before it reaches the store.

**Non-goals.** No feature-level (`@standard`) granularity. No enforcement engine —
standards remain prose the agent reasons over. No automatic re-derivation of links for
changes archived before this feature (the store backfills only as changes are
re-archived).
