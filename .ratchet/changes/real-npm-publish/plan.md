# real-npm-publish

## Why

The `ci-npx-release` batch publishes `ratchet-ai` to npm only on a green build of `main`. The whole publish path has been proven END TO END as a safe dry-run: `gated-publish-job` promoted the publish into its own `publish` job gated on `needs.ci.outputs.release_allowed == 'true'` (the unit-tested `decideRelease` verdict — ALLOW only on `main` with every wired gate green), and `idempotent-version-guard` added a `version-guard` step whose `should_publish` output gates the publish step so an already-published version is a green SKIP rather than a hard E409. In both, the publish step was `npm publish --dry-run`: nothing was ever released, and there was no npm token or provenance permission.

This **real-npm-publish** slice — the final change in the phase — flips that proven dry-run into a REAL release. On a green push to `main`, CI actually publishes the package to the npm registry (with provenance) so end users can run the freshly published CLI via npx. Every gate from the earlier phases still strictly governs the publish: a red lint, test, coverage, e2e, or security signal, or any non-main branch, must still publish nothing.

## What Changes

A thin end-to-end slice that swaps a data SOURCE and the publish COMMAND — not any decision logic — and proves the real publish against a staged registry:

- **Flip the publish step to a real provenance publish.** In `.github/workflows/ci.yml`, change the `publish` job's publish step from `npm publish --dry-run` to a real `npm publish --provenance --access public`, authenticated by an `NPM_TOKEN` repository secret (`NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}`, with `actions/setup-node` configured with `registry-url: https://registry.npmjs.org` so the token is honored). Both fail-closed gates are kept exactly as they are: the job still `needs: [ci]` and runs only when `needs.ci.outputs.release_allowed == 'true'`, and the real publish step still runs only when `steps.<version-guard>.outputs.should_publish == 'true'`.

- **Grant provenance permissions.** Add a job-level `permissions:` block to the `publish` job granting `id-token: write` (so npm can mint a provenance attestation) and `contents: read` (kept minimal). No other permission is widened.

- **Source the already-published set from the real registry.** In `src/core/ci/version-guard.ts`, replace the env-forced published-version source with a real registry query — `npm view ratchet-ai versions --json` — behind an injectable seam so it stays unit-testable offline. The `PUBLISHED_VERSIONS` env override is PRESERVED and takes precedence when set (so tests and the staged-registry proof remain deterministic). A "package not found" (E404) result resolves to an empty set so the FIRST release publishes; any other registry failure fails SAFE toward SKIP (no publish) while still exiting 0, so an ambiguous registry error never republishes and never errors the pipeline. The pure `decidePublishVersion` and `decideRelease` modules are UNCHANGED.

- **Update the structural workflow tests.** The structural assertions that currently require `npm publish --dry-run` and forbid a bare `npm publish` (in `test/ci/idempotent-version-guard.test.ts`) are now inverted by this slice; add/adjust a `test/ci/real-npm-publish.test.ts` (using the workflow helper) asserting the publish step is a real `npm publish` with `--provenance` and `--access public` and no `--dry-run`, the `NODE_AUTH_TOKEN`/`secrets.NPM_TOKEN` wiring, the `id-token: write` permission, and that BOTH gates (`release_allowed == 'true'` and `should_publish == 'true'`) still guard it.

- **Unit-test the real registry source.** Extend `test/ci/version-guard.test.ts` (or a sibling) so the registry query is exercised via the injectable seam: the `PUBLISHED_VERSIONS` override still wins; a successful query populates the set; an E404 yields an empty set (first version PUBLISHes); a non-404 failure yields a SKIP that still exits 0. No test hits the network.

- **Thicken the phase proof harness to a real, staged publish.** Extend `test/e2e/npx-publish.sh` so, on the ALLOW + `should_publish=true` path, it stands up a STAGED local registry (e.g. verdaccio), performs a REAL `npm publish` to it, then runs `npx ratchet-ai --version` against that registry and asserts it EXECUTES the published `ratchet` CLI and prints the published version. It then re-runs the version guard with its real registry source pointed at the staged registry and asserts the just-published version is now seen as already-published → `should_publish=false` → a green idempotent SKIP. The existing gating cases (forced red gate / non-main → no publish, nothing uploaded) are preserved, and the staged registry is torn down so the proof is side-effect-free against real npm.

## Design

**Swap the source and the command, never the decision.** Every earlier slice held to one seam: a pure decision module plus a thin impure runner that gathers its inputs. This slice honors that exactly. `decideRelease` and `decidePublishVersion` are untouched; what changes is (1) where the version guard gets its already-published set (forced env → real `npm view`, with the env override kept as the deterministic test seam) and (2) the literal publish command in the workflow (`--dry-run` → real provenance publish). The "only when green on main" and "PUBLISH vs SKIP" logic that the batch spent three phases proving is reused verbatim.

**Two fail-closed gates, unchanged, now guarding a real upload.** The publish path still passes both gates in series — the job-level release gate (`release_allowed == 'true'`, governs whether the `publish` job runs at all; fail-CLOSED off main / on any red gate) and the step-level version guard (`should_publish == 'true'`, governs whether it actually publishes; fail-SAFE toward green on a SKIP but fail-CLOSED toward publishing). Flipping to a real publish does not relax either; it only changes what happens once both are satisfied. Nothing publishes unless explicitly permitted AND explicitly new.

**Provenance needs `id-token: write`.** npm provenance mints a signed attestation via GitHub's OIDC token, which requires the job to hold `id-token: write`. Granting it at the `publish` job level (with `contents: read`) keeps the elevated permission scoped to the one job that publishes, rather than the whole workflow — least privilege.

**First publish and registry failures, handled deliberately.** A brand-new package has no versions, and `npm view` returns E404 for it; treating E404 as an empty set lets the very first release publish instead of being mistaken for an error. Any OTHER registry failure is genuinely ambiguous (the version MIGHT already be published), so the guard fails SAFE toward SKIP — it does not publish, but it still exits 0, preserving idempotency and never reddening the pipeline on a flaky registry. These two branches live in the impure runner; the pure decision still only classifies a known set.

**Staged registry for a deterministic real proof.** Actually uploading to npmjs.org from a test would be non-deterministic and irreversible, so the blackbox proof publishes to a local staged registry (verdaccio) — which the phase proof-of-work explicitly permits ("real (or staged-registry) publish"). This exercises the genuine `npm publish` and `npx` codepaths (pack, manifest, registry negotiation, install, bin resolution) end to end, then asserts `npx ratchet-ai --version` runs the PUBLISHED CLI — the real user-facing guarantee — without touching the public registry. Branch and gate signals stay forced via environment so gating remains deterministic.

**Naming: `ratchet-ai` package, `ratchet` bin.** The package was renamed to `ratchet-ai` while keeping the `ratchet` bin, so the registry query targets `ratchet-ai` and the npx assertion is `npx ratchet-ai --version` (npx resolves the `ratchet-ai` package and runs its `ratchet` bin). The phase's shorthand "`npx ratchet --version`" maps to this exact invocation.

**Scope discipline.** This slice changes only the publish SOURCE/COMMAND and proves it: the registry query, the real provenance publish + token + `id-token` permission, and the staged-registry/`npx` proof. It does not alter `decideRelease`, `WIRED_GATES`, `decidePublishVersion`, the gate plumbing, or any earlier gate (lint/test/coverage/e2e/security). It does not bump the package version or add tag handling beyond what `npm publish` reads from `package.json` (the idempotent guard already makes a re-run of an unchanged version a safe SKIP).

**Trade-offs.** A staged registry means the proof does not exercise the literal npmjs.org upload or the OIDC provenance signing (which needs the real Actions OIDC issuer) — accepted, because those require live GitHub/npm infrastructure and are wired structurally in the workflow and asserted by the workflow test; the staged publish proves every codepath under our control. Failing SAFE toward SKIP on an ambiguous registry error trades a possible missed publish (recoverable on the next green push) for never double-publishing or erroring on a flaky registry — the right bias for an idempotent release.
