# secret-scan-gate

## Why

The `ci-npx-release` batch publishes `ratchet` to npm only when every quality gate is green, and its promise is that "only when green" is a **real, unit-tested decision**, not just YAML wiring. Phase 1 shipped the dry-run pipeline skeleton plus the pure, unit-tested `decideRelease` spine and the thin `release-gate` runner (`src/core/ci/release-decision.ts`, `src/core/ci/release-gate.ts`). Phase 2 added the **coverage** and **e2e** gates the same way — a pure evaluator + thin runner per slice (`coverage-gate.ts`, `e2e-gate.ts`), then a wiring change that plugged both into the spine.

Phase 3 (`security-layer-gate`) adds a **security layer** as a hard gate: a CI job that runs a dependency vulnerability audit and a secret scan (and prepares npm provenance), wired into the release-decision module so a known vulnerability or a leaked secret flips the gate to **DENY**. That phase is sliced into three changes: `dependency-audit-gate` (already landed), this one (`secret-scan-gate`), and `wire-security-into-release-gate` (which runs `after` both and plugs their signal(s) into the spine and ships the phase proof `test/e2e/security-gate.sh`).

This change is the **secret-scan slice**: a pure, exhaustively unit-tested evaluator that turns a parsed secret-scan report into a `green`/`red` **gate signal**, plus the CI wiring that runs the scan and goes red when a finding is present. It deliberately mirrors the sibling `dependency-audit-gate` split — a pure module + a thin runner + a minimal CI step — and **does not** wire the signal into `decideRelease`'s wired-gate set. That wiring belongs to `wire-security-into-release-gate`. This slice exists to prove the "no leaked secret" decision in isolation so the wiring change has a trustworthy signal to feed in.

## What Changes

- **Pure secret-scan evaluator.** Add a shippable module under `src/core/ci/` (`src/core/ci/secret-scan-gate.ts`) exporting a pure `evaluateSecretScan(input)` that takes a parsed secret-scan report (a list of findings, e.g. `{ rule, file, line, secretType }[]`) plus an optional allowlist, and returns an explicit `{ signal: 'green' | 'red'; findings: SecretFinding[] | null; allowlisted: number; reasons: string[] }`. `green` iff the report parsed (non-null) AND **zero** non-allowlisted findings remain; otherwise `red` with a precise reason naming each offending finding (file + rule), or "secret-scan report could not be read" when the findings are missing/malformed. The returned `signal` is exactly the `GateSignal` shape (`'green' | 'red'`) the release-decision module already consumes, so the later wiring change feeds it in unchanged.
- **Allowlist as data.** Define an allowlist concept (e.g. known-safe placeholders / fixture secrets used by tests) resolvable from an env var (e.g. `SECRET_SCAN_ALLOWLIST`), so an intentionally-planted fixture secret in the test suite does not turn a real run red, while a genuine leak does. Keep the allowlist data, not a hardcoded literal at the call site. Default to an empty allowlist — any finding is red unless explicitly allowlisted.
- **Scan report parsing, fail-closed.** Add a small reader that extracts the findings array from the secret scanner's machine-readable output (a JSON report such as gitleaks' `--report-format json`, or an equivalent regex-scan emitter shipped here). A missing, unreadable, or malformed report yields `null`, which the evaluator treats as `red` — fail-closed, matching the spine's posture and the sibling dependency-audit slice.
- **Thin secret-scan-gate runner.** Add a thin, impure entrypoint (`runSecretScanGate(env)` in the same module, run when invoked directly à la the sibling gates' `isDirectRun()`) that resolves the report path + allowlist from the environment, reads the findings, calls `evaluateSecretScan`, prints the outcome and any reasons, and exits `0` on `green` / non-zero on `red`. It adds **no** decision logic — it only adapts CI's world to the pure evaluator, exactly as `release-gate.ts` / `coverage-gate.ts` / `e2e-gate.ts` / `dependency-audit-gate.ts` do.
- **CI step.** Add a secret-scan step to `.github/workflows/ci.yml`, positioned **after** the existing `test` step and **before/outside** the release path: run the secret scan writing its JSON report (e.g. a scanner invocation `... --report-format json --report-path <report> || true`, tolerating the scanner's own non-zero exit so the gate runner — not the raw scanner — owns the red/green decision), then `pnpm build` (if needed) and invoke the built secret-scan-gate runner (`node dist/core/ci/secret-scan-gate.js`) so a finding exits non-zero and turns the run red. Give the step an `id` (e.g. `secret-scan`) the later wiring change can read. Do **not** add a security signal to the release-gate step and do **not** extend `WIRED_GATES` — that is `wire-security-into-release-gate`'s job.
- **Tests** under `test/ci/` (so they run with the suite):
  - `test/ci/secret-scan-gate.test.ts`: `green` on zero findings; `red` when a finding is present (reason names the file/rule); `red` accounting for several findings; fail-closed `red` on a missing/malformed report (reason names the unreadable report); allowlist respected (an allowlisted finding lets a tree stay green); and the returned `signal` is assignable to the release-decision `GateSignal` type.
  - Secret-scan CI-step assertions (using the existing `test/ci/helpers/workflow.ts` parser): a step runs the secret scan and writes a JSON report; a secret-scan-gate step invokes the evaluator and sits after the `test` step; and the release path still wires only `lint`/`test`/`coverage`/`e2e` (no security signal yet).
- Implements `features/ci/secret-scan-gate.feature` and `features/ci/secret-scan-ci-step.feature`.
- **Out of scope (explicitly):** the dependency audit (`dependency-audit-gate`, already landed), npm provenance preparation, wiring the security signal into `decideRelease`, and the phase proof harness `test/e2e/security-gate.sh` — all belong to the sibling changes (`wire-security-into-release-gate`). The publish path remains `--dry-run`.
- Multi-agent surface: **none**. A pure evaluator, a thin runner, workflow YAML, and tests; no agent-specific files, skills, or commands.

## Design

**Thin vertical slice, same spine.** The phase goal is "a known vulnerability OR a leaked secret flips the gate to DENY". The full proof (`bash test/e2e/security-gate.sh`) spans all three phase-3 changes. This change owns the smallest end-to-end secret-scan piece: run the scan → parse the report → decide green/red on any finding → emit a gate signal in the spine's shape, with CI going red on a planted secret. Joining that signal to `decideRelease` (extending `WIRED_GATES`, feeding the security signal) is the explicit `after` change — kept out of scope here exactly as the sibling `dependency-audit-gate` kept its wiring out.

**Pure evaluator + thin runner, mirroring the sibling gate.** The risky, must-be-correct part is "is there any non-allowlisted finding, fail-closed?". That is isolated as a pure function so every branch is unit-testable without running a real scan, precisely as `evaluateDependencyAudit` / `evaluateCoverage` / `evaluateE2e` isolated their decisions. The runner is impure glue — read the report + allowlist from env, translate the verdict into an exit code — and is exercised directly with a fixture report, no Actions runner needed. This keeps the security gate provable, not hoped.

**Signal shape is the integration contract.** `evaluateSecretScan` returns `signal: 'green' | 'red'` — the same `GateSignal` the release-decision module keys its gates by. That is the seam the wiring change snaps into: it will combine this slice's signal (and the dependency-audit slice's) into the single security signal the release-gate step passes to `decideRelease`, with no change to the core logic, which already treats gates as data. Pinning the shape now (and asserting type-compatibility in tests) is what makes the later wiring a small addition.

**Allowlist, fail-closed on the unknown.** Any finding is red by default — a secret gate must err toward blocking. The allowlist exists only to exempt intentionally-planted fixture secrets (the test suite plants one to prove the gate trips) and known-safe placeholders, and it is data (`SECRET_SCAN_ALLOWLIST`) so it can be adjusted without touching code. Conversely, a missing or malformed report is treated as `red`: the safe posture for a security gate, asserted directly — a scan that never ran (or crashed before writing its report) must never read as green.

**Gate runner owns the decision, not the raw scanner exit.** A scanner's own exit code is too blunt to gate a release on (it may exit non-zero on warnings or zero despite a finding, depending on flags). So the CI step tolerates the scanner's own exit (`|| true`) and lets the unit-tested gate runner decide red/green from the parsed report against the allowlist. Enforcement lives in tested code, not in the scanner tool's default exit — matching the batch's thesis that the gate is provable code, not a tool's default hoped to be correct.

**Structure asserted via the parser, behavior via the runner.** A unit test cannot run GitHub Actions, so the CI half is proven structurally against the parsed model the parser helper exposes: a secret-scan step that writes a JSON report, a secret-scan-gate step after the `test` step, and the absence of any security signal on the release path (the boundary). The decision half is proven behaviorally by exercising the evaluator and runner directly with fixture reports. The real red/green behavior runs for real on every push, which the batch wants observable.

**Trade-offs.** Splitting "produce the secret-scan signal" (here) from "consult it in the release decision" (the `after` change) is deliberate: it keeps each slice independently provable and the release-decision spine untouched until both phase-3 signals (audit + secret scan) exist, so they are wired together once rather than the module churning twice. The cost is that the full phase proof (`security-gate.sh`) only goes green after all three changes land; this slice's own features and unit tests stand on their own in the meantime.
