# `.ratchet/evals/` — the eval suite

This directory turns ratchet's Gherkin `.feature` files into a **scored,
reproducible, baseline-diffed regression suite**. Cases are enumerated from the
feature files; this directory holds the data that says *how to judge each one*
and *what codebase to judge it against*.

```
.ratchet/evals/
  specs/      eval-spec YAML — binds a case id to a fixture + a judging check
  fixtures/   checked-in, pre-determined codebases each case runs against
  runs/       persisted run results (one JSON per `ratchet eval run`)
  baseline.json   (optional) pointer to the run future runs are diffed against
```

## How it works

1. `ratchet eval set` enumerates **one case per Scenario** from
   `.ratchet/features/**` (the store) and, with scope flags, from active changes.
   Each case has a stable id: `<feature-path-sans-ext>#<scenario-slug>`.
2. An **eval-spec** under `specs/` binds a case id to a `fixture` and a judge:
   - `kind: deterministic` — a deterministic bash command run against the fixture,
     with a `pass` condition (`exit-zero` | `contains:<text>` | `regex:<pattern>` |
     substring). No agent; same answer every time.
   - `kind: llm-judge` — a fresh coding agent is spawned against the fixture and
     judged against `success` criteria (fail-closed on uncertainty; a configurable
     `jury` — `votes`, default 1, layered from the project's `eval.jury` default;
     `quorum`, `majority` (default) or `unanimous` — resolves the cast votes, and a
     sub-quorum result is `unjudged`). For prose-y scenarios.
   A case with no binding is **`unjudged`** — never counted as a pass.
3. At judge time each case gets its **own throwaway copy** of the fixture as its
   working directory (see `src/core/eval/fixture.ts`), so a check may freely
   mutate it (e.g. archiving moves directories) without affecting other cases or
   the checked-in fixture. An optional one-time `setup` bootstraps a fixture once
   (cached, reused across cases).
4. `ratchet eval run` judges every bound in-scope case and persists a run to
   `runs/`. `ratchet eval report --run <id>` prints the scorecard and diffs
   against the baseline (a case that was `pass` in the baseline and `fail` now is a
   **regression**; new/retired cases are diffed, not failed). `ratchet eval
   baseline <run-id>` promotes a clean run.

**Fixtures never pollute enumeration.** `ratchet eval set` only scopes to
`.ratchet/features/**` and `.ratchet/changes/**`; `.ratchet/evals/fixtures/` is
never a scope root, so the fixtures' own nested `.ratchet/` projects are not
enumerated as cases. (Verified: store and `--changes` case counts are unchanged
by the fixtures here.)

## Common commands

```bash
ratchet eval set  --path <dir-or-file> --json     # list cases (+ binding status)
ratchet eval run  --path archive-change --judge deterministic   # judge bound cases, persist a run
ratchet eval report  --run <run-id>               # scorecard + baseline diff
ratchet eval baseline <run-id>                    # lock in a clean run as the baseline
```

`--judge auto` (default) judges each case by its bound `kind`; `--judge deterministic`
runs only deterministic checks; `--judge llm-judge` forces the spawned-agent judge.

## Worked example: the `archive-change` eval

The `archive-change` feature (`.ratchet/features/archive-change/archive.feature`,
6 scenarios) is fully covered by `specs/archive-change.yaml` — all six bindings
are `deterministic`, and `ratchet eval run --path archive-change` reports
**6 pass / 0 fail / 0 unjudged**.

### Where the ratchet CLI is exercised

This suite runs the real CLI at **three layers** — it is not just static fixture
files:

1. **Building the fixtures (one-time authoring).** The fixtures were not
   hand-written blind. They were generated by running ratchet itself in a scratch
   directory — `ratchet init`, `ratchet new change add-login`, then authoring the
   feature + plan — and by reading real output of `ratchet archive --help` /
   `ratchet archive` to capture the exact, asserted-on messages (e.g.
   `Validation errors in change features`, `Plan warnings in plan.md
   (non-blocking)`, `1 incomplete task(s) found`, `already exists`). The resulting
   project tree was copied in as the fixture, so each fixture matches ratchet's
   real on-disk expectations.
2. **Inside every check, at judge time.** Each binding's `run` script invokes the
   real CLI against the fixture's throwaway working copy — `ratchet archive
   add-login …` (and the twice-same-day case also runs `ratchet new change
   add-login` to recreate the collision) — then asserts the scenario's `Then`
   post-condition on the filesystem/stdout, exiting non-zero on any mismatch.
3. **Driving the suite.** `ratchet eval set` / `run` / `report` enumerate, judge,
   and score.

### Why `pass: exit-zero` is sound here

`ratchet archive` exits **0 even when it blocks** an archive (it prints an error
and returns without moving the change). So an exit code alone cannot tell pass
from fail. Each `run` script therefore asserts the actual outcome — whether the
change directory was moved into `.ratchet/changes/archive/<date>-add-login`,
whether the feature was synced into `.ratchet/features/`, whether the store was
left untouched — and only then does `exit-zero` faithfully mean "scenario held".

### The four fixtures

Each is a minimal nested ratchet project (`.ratchet/config.yaml` + an active
`add-login` change: `.ratchet.yaml`, `plan.md`, `README.md`,
`features/auth/login.feature`). No `node_modules`, no `setup` step. They differ
only in the precondition:

| Fixture | State | Used by |
|---|---|---|
| `archive-ready` | valid feature + complete plan (all tasks `[x]`) | complete-syncs-and-moves, skip-feature-store, twice-same-day |
| `archive-feature-error` | feature scenario missing a `When` step | feature-errors-block |
| `archive-plan-warning` | valid feature + `## Why` too short → WARNING | plan-warnings-non-blocking |
| `archive-incomplete` | valid feature + one unchecked task | incomplete-tasks-confirmation |

### Per-scenario check strategy

| Case (slug) | Fixture | What the `run` script does |
|---|---|---|
| `archiving-a-complete-change-syncs-features-and-moves-the-directory` | archive-ready | `archive -y`; assert feature synced into store **and** change moved to dated archive |
| `skipping-the-feature-store-leaves-it-untouched` | archive-ready | `archive -y --skip-features`; assert store file never created **and** change still archived |
| `feature-errors-block-the-archive` | archive-feature-error | `archive`; grep validation-error output; assert change still active, no dated archive |
| `plan-warnings-do-not-block-the-archive` | archive-plan-warning | `archive -y`; grep non-blocking notice; assert archive proceeded |
| `incomplete-tasks-require-confirmation` | archive-incomplete | `archive` (no `-y`); stdin is closed so the confirm prompt gets EOF and declines; grep the incomplete-task warning; assert change NOT moved |
| `archiving-twice-on-the-same-day-fails` | archive-ready | archive once; `ratchet new change add-login`; archive again; grep `already exists`; assert first archive byte-identical (preserved) and new change not moved over it |

### Re-running / locking it in

```bash
ratchet eval run --path archive-change --judge deterministic   # → 6 pass
ratchet eval baseline <run-id>                         # lock as baseline
```

Once a baseline is promoted, any future change that breaks archive behavior will
surface as a **regression** in `ratchet eval report`.

## Authoring a new eval (recipe)

1. `ratchet eval set --path <feature-dir> --json` to get the exact case ids.
2. Create a fixture under `fixtures/<name>/` — generate it by running ratchet in a
   scratch dir for realism, then copy it in. Keep it minimal and self-contained.
3. Add a binding per case in a `specs/*.yaml` file (top-level `caseId: {…}` or
   under a `bindings:` key). Prefer `kind: deterministic` with a self-asserting
   `run` script; fall back to `kind: llm-judge` + `success` only for genuinely
   prose-y scenarios.
4. Iterate `ratchet eval run --path <dir> --judge deterministic` until green.
5. Verify `ratchet eval set` (store / `--changes`) case counts are unchanged by
   your fixtures.

## LLM-judged fixtures need a tool allowlist

For `kind: llm-judge` bindings, the judge is a freshly spawned coding-agent
subprocess whose cwd is the throwaway fixture copy. `FixtureManager` copies the
whole fixture recursively, so a **fixture-local agent config travels into that
copy** and is what grants the spawned judge permission to act.

- For a Claude judge, put `.claude/settings.json` in the fixture with a tool
  allowlist (e.g. `Bash, Read, Glob, Grep, LS`). Without it the sandboxed judge
  cannot run commands and **fails closed** (correct, but the case never passes).
  Use a scoped allowlist, not blanket bypass.
- The repo's `.gitignore` ignores `.claude/`, so a fixture's `.claude/settings.json`
  must be force-added (`git add -f`) to be tracked.
- Adapter selection: `ratchet eval run` currently always resolves the default
  adapter (`claude`) — there is no flag to pick another agent (only the
  `RATCHET_EVAL_AGENT_CMD` test stub). Selecting the adapter is a possible
  follow-up in the eval implementation.

See `fixtures/eval-self-run/` — an llm-judged case that dogfoods eval itself
(its inner binding is `deterministic`, so the recursion depth is 1).

## Driving `ratchet batch apply` deterministically (the agent stub seam)

`ratchet batch apply` normally spawns a real coding agent to drive a transition,
which a `kind: deterministic` binding cannot do reproducibly. The **batch-agent-stub-seam** adds
an env override mirroring `RATCHET_EVAL_AGENT_CMD`: when `RATCHET_BATCH_AGENT_CMD`
is set, the engine runs that command via `bash -c` (feeding the step instructions
on stdin) **instead of** resolving/spawning the configured adapter; unset → the
adapter path is unchanged. The unit-test `Spawner` injection is untouched and
complementary — the env var is the CLI/e2e/eval seam.

- Fixture `fixtures/batch-apply/` is a self-contained `q3-auth` batch with one
  ready change plus two checked-in scripted "agents":
  - `agent.sh` — reads the instructions on stdin, learns the transition
    (PROPOSE/APPLY/VERIFY) and the batch/change from the report-channel line,
    then drives that transition for real (scaffolds a `plan.md` + `features/` on
    propose, checks the task boxes on apply) and reports via `ratchet batch
    report`.
  - `agent-blocker.sh` — raises a blocker on the first propose to park the step,
    then proceeds normally once an answer is in context (the resume path).
- A batch-orchestration check points the seam at the script and asserts the
  observable state, e.g. `export RATCHET_BATCH_AGENT_CMD="bash $PWD/agent.sh"`
  then `ratchet batch apply q3-auth --json` and a `python3 -c` assertion on
  `ratchet batch status --json`. The fixture copy is the cwd, so `$PWD/agent.sh`
  resolves inside it.

**Deferred host-loop gap (kept honest, unjudged):** the *verify* transition and
proof-of-work phase gating are not reachable through `batch apply` today — once a
change's plan tasks are all checked it is `done` and the step-picker treats done
as terminal, and `runProofOfWork` is not wired into `runStep`. The
`batch-apply#...propose-then-apply-then-verify` and the `batch-phases` proof-of-
work cases therefore stay `unjudged` (documented in `specs/batch-orchestration.yaml`)
until the host loop lands.
